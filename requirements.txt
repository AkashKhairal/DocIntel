You are an expert AI engineer and full-stack software architect.

Build a **production-ready Enterprise Document Intelligence System** that allows employees to chat with company documents stored in Google Drive.

The system must support **event-driven indexing using Google Drive webhooks**, so when a new file is uploaded or modified the system automatically processes it.

The system must **not periodically poll Google Drive**. It must instead use **Google Drive push notifications (watch channels)**.

The system must support **50GB+ documents** and follow a **Retrieval Augmented Generation (RAG)** architecture.

---

# SYSTEM OBJECTIVE

Employees should be able to ask questions such as:

"What work did Rahul do in February?"

"Which documents were created for client HDFC?"

"Show me all reports created in March."

The system must answer only using company documents and provide **citations for the files used**.

If the information does not exist in the documents, the system must respond:

"I could not find this information in the company documents."

No hallucinations should occur.

---

# CORE FEATURES

The system must include:

• Google Drive connector
• Event-driven document ingestion
• Automatic indexing of new/modified files
• Document chunking and embedding
• Vector database storage
• Chat interface for document Q&A
• Source citation
• Minimal modern frontend

---

The architecture should follow this flow:

# GOOGLE DRIVE INTEGRATION

Use the **Google Drive API v3**.

Implement **Push Notifications (Watch Channels)**.

When a file is:

• created
• updated
• deleted

Google Drive will send a notification to the system webhook endpoint.

Create an API endpoint:

POST /drive/webhook

The webhook handler must:

1. Receive Drive notification
2. Fetch the change using the Drive Changes API
3. Identify which file changed
4. Push file processing job to background worker

---

# DOCUMENT INGESTION WORKFLOW

When a new file or modification is detected:

1. Download the file temporarily
2. Extract text from the document
3. Split document into chunks
4. Generate embeddings
5. Store embeddings in Qdrant
6. Store metadata
7. Delete the temporary file

Important:
The system must **never permanently store the entire document locally**.

---

# METADATA STRUCTURE

Each document chunk should store metadata:

file_name
drive_file_id
drive_link
folder_path
created_time
modified_time

---

# CHUNKING

Chunk size: 500 tokens
Overlap: 100 tokens

---

# EMBEDDING GENERATION

Use:

BAAI/bge-large-en-v1.5

Generate embeddings using HuggingFace.

Store embeddings in Qdrant.

---

# RETRIEVAL PIPELINE

When a user asks a question:

1. Convert the query into an embedding
2. Perform vector search in Qdrant
3. Retrieve top 10 relevant chunks
4. Send retrieved chunks to LLM
5. Generate grounded response

---

# ANTI-HALLUCINATION PROMPT

The LLM must follow this instruction:

"You must answer ONLY using the provided context. If the answer is not present in the documents, say that it cannot be found."
